# public surface loop

## Why this exists
A public surface should make the right people understand you faster.

The point is not self-display.
The point is to lower the cost of discovery, interpretation, and contact.

## Order of operations
1. decide the core axis
2. collect public proof
3. choose what each surface must do
4. write minimal copy
5. remove misleading signals
6. update only when real evidence changes

## 1. decide the core axis
Write one line that answers:
- what kind of builder are you
- what kind of problems do you actually work on
- what should a stranger remember after 10 seconds

Good shape:
- LLM systems, developer tooling, and research engineering
- compiler/runtime work, tooling, and evaluation infrastructure

Bad shape:
- curious multidisciplinary technologist
- passionate builder at the intersection of everything

## 2. collect public proof
Prefer 3-5 links only.

Good proof:
- strong repo with real history
- profile README
- public resume
- paper / doc / benchmark artifact
- talk or demo

Weak proof:
- repo with no real substance
- vague project page
- writing with no underlying work

## 3. choose the job of each surface
### LinkedIn
Job:
- be found
- be understood quickly
- route people to stronger evidence
- allow contact

### GitHub
Job:
- prove capability
- show taste in project selection
- show depth, not just activity

### X
Job:
- optional distribution layer
- optional weak-tie layer
- optional OSS visibility layer

If it does not serve a real loop, do not force it.

## 4. write minimal copy
### LinkedIn headline
Should say:
- role axis
- problem axis
- maybe domain if it helps

Example shape:
- LLM systems | developer tooling | research engineering

### LinkedIn About
Should do four things:
- what you build
- what kind of problems you care about
- what evidence exists
- what kinds of roles or conversations fit

### Featured section
Prefer links that make the next click obvious:
- GitHub profile
- one or two flagship repos
- resume
- one public doc or paper if strong

## 5. remove misleading signals
Cut things that:
- overstate seniority
- suggest a different axis than your best work
- invite the wrong recruiter traffic
- create a weaker story than the underlying evidence

## 6. decide what is worth posting
Post only when one of these is true:
- a meaningful public repo milestone shipped
- a real artifact became public
- you have a short technical judgment backed by work
- you learned something from building, measuring, or comparing

Do not post because a calendar says to.

## Default cadence
- LinkedIn profile text: update when the story changed
- Featured links: update when better proof exists
- X or LinkedIn posts: low frequency, high signal

## Quick audit questions
- if someone reads only the headline, do they get the right picture
- if someone clicks one link, does the story get stronger
- if someone clicks three links, does the story remain consistent
- does every featured item deserve scarce attention
- would this surface still look honest if the viewer were skeptical
