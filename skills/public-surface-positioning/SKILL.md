# public-surface-positioning

Use when the user already has real work, real artifacts, or real career evidence and wants to make their public surface easier to read.

## Goal
Turn evidence into a clearer public-facing position.

## This is for
- clarifying a LinkedIn headline and About section
- deciding what GitHub should lead with
- deciding whether X should exist at all
- choosing which links and projects deserve public emphasis

## This is not for
- fake confidence
- empty positioning language
- growth-hacking
- posting for its own sake
- pretending vague ambition is evidence

## Inputs
- `profile.md`
- `work-model.md`
- `targets.md`
- public repos, resume, portfolio, papers, talks, or docs

## Output
Produce:
- core professional axis in one line
- recruiter-facing read in one line
- hiring-manager-facing read in one line
- 3-5 proof links
- LinkedIn headline candidates
- LinkedIn About bullets or short paragraph
- GitHub top-line positioning
- optional X stance: none / light presence / active
- do-not-say list
- update triggers and posting cadence

## Rules
- evidence first, wording second
- public surface should reduce interpretation cost, not increase performance pressure
- prefer a small number of strong links over many weak claims
- if a repo would embarrass the story when clicked, do not feature it
- use the same core axis across LinkedIn, GitHub, and resume unless there is a role-specific reason to split
- treat LinkedIn as identity page + routing page + contact page
- treat X as optional; default to no or light presence unless it serves a concrete loop

## Minimal recipe
1. name the core axis in plain language
2. list the strongest public proof
3. decide what each surface is for
4. write the shortest wording that routes attention correctly
5. remove anything that creates the wrong first impression
6. define what kinds of updates are actually worth posting

## Surface policy
### LinkedIn
Optimize for:
- searchability
- quick legibility
- easy contact
- clean routing to stronger evidence

### GitHub
Optimize for:
- proof of work
- project selection
- clear top-line technical identity

### X
Use only if it helps with:
- OSS visibility
- weak-tie discovery
- participation in a real technical conversation

Default stance: low-frequency, high-signal, evidence-backed updates.

## Good prompts to run through this skill
- Turn my actual work into a LinkedIn headline and About without sounding inflated.
- Decide what my GitHub profile should lead with for infra / tooling roles.
- Decide whether I should maintain X at all.
- Build a public surface from these repos and resume bullets.
